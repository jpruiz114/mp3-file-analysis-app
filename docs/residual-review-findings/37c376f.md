## Accepted residual: CPU-exhaustion resistance is partial, not complete

**Reported independently by two reviewers:**

- `security` (P1, confidence 75): benchmarked adversarial near-miss-sync input
  (bytes that pass the cheap 11-bit sync check but fail deeper header validation)
  costing ~46x more CPU per byte than real audio content, on an unauthenticated
  public endpoint with no rate limiting.
- `adversarial` (P1, confidence 75): benchmarked a related amplification vector —
  forwarding every network `'data'` chunk 1:1 into `FrameCounter.write()` with no
  coalescing costs ~50-150x more CPU end-to-end when an attacker controls TCP write
  granularity (many tiny chunks).

**What was fixed (commit `37c376f`):** `src/mp3/frameHeader.ts`'s `parseFrameHeader`
now returns shared singleton objects for rejections instead of allocating a fresh
object per call, and only reads the header bytes it needs before the cheap sync
check runs. Verified ~2.6x faster on 20MB of the exact adversarial pattern
benchmarked, with identical output — a real, zero-risk reduction in the per-byte
constant factor.

**What remains open:** the underlying cost is still O(n) synchronous CPU work with
no per-request time/CPU budget, and the chunk-granularity amplification vector
(adversarial's finding) was not restructured. A full mitigation would require one or
more of: a per-request CPU/wall-clock budget that aborts on overrun, offloading the
parse loop to a worker thread, minimum-chunk coalescing before processing, or
rate limiting / concurrency caps in front of the endpoint.

**Why accepted rather than fixed:** the assignment's own plan (`docs/plans/2026-07-07-001-feat-mp3-frame-upload-api-plan.md`, Scope Boundaries) explicitly scoped
authentication, authorization, and rate limiting out of this exercise before any of
this review ran — this residual is squarely inside that boundary, not a new gap
introduced by shipping late. Building request-throttling infrastructure or a worker-
thread pool under take-home time constraints, for a concern normally handled at a
reverse-proxy/gateway layer in production, was judged disproportionate to fix now
versus fixing the actual free, verified, zero-risk win available (the singleton-
allocation fix, which was applied).

Documented in `README.md` under "What I'd do with more time" as well.
